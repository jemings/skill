# skill-optimizer — SKILL.md 슬림화·구조 최적화

## 원칙

스킬이 트리거되면 SKILL.md 본문 전체가 컨텍스트에 적재되고, frontmatter
`description` 은 트리거와 무관하게 매 세션 스킬 목록에 상주한다 — 군더더기가 곧
토큰 비용(description 은 매 세션). 절대 제약은 **동작·트리거 커버리지·인바운드
앵커·정보 불변** — 보존하는 것은 효과이지 문장이 아니다(트리거도 phrasing 이
아니라 커버리지). 단, 불변 대상은 **올바른** 정보다: SSOT 대조에서 드러난 사실
오류·누락·깨진 마크업은 보존이 아니라 수정 대상. 같은 효과를 더 적은 토큰으로
내는 다섯 lever — **description 슬림화 · prose 압축 · 레퍼런스 분리 · 셸 외부화 ·
훅 흡수**(각 적용·주의는 3단계).

핵심 위험은 load-bearing 요소(1단계 불변식 목록)를 무심코 압축해 스킬
매칭/링크를 **조용히** 깨뜨리는 것 — 그래서 아래 4단계 순서를 강제한다.

## 1단계 — 측정 & 불변식 발견 (read-only, 편집 전 필수)

베이스라인: `wc -lc <SKILL.md>` + 토큰 추정(chars/4; 패딩 공백 많은 표는
`tr -s ' '` 후 재측정).

**사본 토폴로지** — 같은 스킬이 여러 위치에 있는지 먼저 확인한다(프로젝트
`.claude/skills/` · top-level `skills/` · 사용자 `~/.claude/skills/` ·
마켓플레이스 repo). 2곳 이상이면 각 위치의 실제 소비자(로더·스캐너·설치 CLI)를
코드로 확인하고 단일화 가능성을 3단계 후보로 올린다 — 수동 동기화는 조용히
drift 하고, 중복 사본은 스캐너 오분류(single→bundle 등)를 유발한다.

불변식 grep — 깨면 조용히 망가지는 것들:

- **frontmatter `description` 트리거 커버리지** — 매칭되는 의도·시나리오 집합 식별
  (예시 발화·키워드가 각각 무엇을 잡는지). 불변식은 커버리지이지 phrasing 이 아님.
- **인바운드 앵커**: 섹션 헤딩을 타 문서가 `#슬러그` 로 링크하는지
  `git grep -n "<heading-slug>" -- '*.md'` (git repo 가 아니면
  `grep -rn --include='*.md'`). 헤딩 텍스트 보존 → 앵커 유지.
- **번호 참조 룰**: 타 문서가 "AI 행동 룰 1" 처럼 번호로 참조하면 순서·번호
  보존.
- **함수/명령 이름·시그니처**: 타 문서·대화에서 그대로 인용 — 불변.
- **사실 주장 SSOT 대조**: 번들 스크립트와 SKILL.md 의 주장(상수 위치 등) 일치
  확인. 스킬이 외부 표면(API·CLI·설정)을 문서화하면 그 시스템의 실제 코드와도
  대조한다(예: 엔드포인트 표·에러 목록·필드 제약 — 범위가 크면 읽기 전용
  서브에이전트로 병렬 탐색).

**무게 진단** — 불변식(트리거 커버리지·앵커 등) 제외 후 남은 무게를 원칙의
다섯 lever 기준으로 분류해 매핑(description 비-트리거·중복 phrasing →
description 슬림화 · 군더더기 prose → 압축 · 트리거 시점에 불필요한 대량
예시·참조 데이터 → 레퍼런스 분리 · 절차 로직 → 셸 외부화 · 이벤트 트리거형
always-run 지시 → 훅 흡수). 레퍼런스 분리 판정은 읽기 경로 기준: 트리거 즉시
필요한 섹션인지, 특정 흐름에서만 읽는지. 애매하면 prose 압축으로 보수적 처리.

산출: 보존 항목 목록 + lever 별 슬림 후보 목록 + 사실 오류 수정 목록.

## 2단계 — 슬림 계획

- 적용 전 사용자가 스캔할 수 있는 **짧은 변경 목록**(보존 항목 / 제거·병합
  항목)을 제시한다. 계획 없는 일괄 편집 금지 — 이 계획이 곧 4단계 검증
  체크리스트다.
- 사본 단일화·공유 파일 변경(해당 스코프 settings.json hooks 등록 등)은 다른
  세션·팀원·설치된 소비자에 영향 — 계획에 별도 항목으로 명시해 사용자 확인을
  받는다.

## 3단계 — 슬림 레시피 적용 (Edit)

- **description 슬림화**: frontmatter `description` 에서 비-트리거 내용(내부
  동작·메커니즘·보장 문구·본문 재진술)은 제거(본문에 없는 실제 정보면 본문으로
  이동)하고, 트리거 phrasing 은 중복·군더더기를 압축한다. 기준은 토큰이 아니라
  커버리지 — distinct 매칭 단서(그것만이 잡는 발화/키워드)는 줄이지 말 것.
- **사실 오류 수정**: 1단계 SSOT 대조에서 드러난 오류·누락·깨진 마크업 반영.
- **출처/메타 제거**: 이슈·PR 번호 등 AI 동작과 무관한 provenance 삭제
  (잔여 확인: `grep -nE "#[0-9]{2,}" <SKILL.md>`). 정책 추적 필요분만 번호 없는
  SSOT 링크로 대체.
- **중복 제거**: 다이어그램·표·산문·룰에 걸쳐 같은 내용은 한 곳에만. 핵심은
  인라인 흡수.
- **표 슬림**: AI 가 직접 호출하는 public 표면만 남기고, 내부 헬퍼·CI·hook 행은
  "세부는 `scripts/...` 참조" 한 줄 포인터로.
- **마이크로 패스**: 얇은 인접 섹션 병합 · 도입/전환 문장 제거 · 괄호 노트 본문
  흡수 · 멀티라인 명령 블록 → 인라인 코드.
- **사본 단일화**: 소비자가 공통 인정하는 위치 하나만 남기고 나머지 삭제.
  삭제 전 남을 경로에 대한 참조를 전수 grep 하고, 옛 경로를 가리키는 라이브
  소비자(설치된 플러그인·마켓플레이스 등록 등)가 있으면 재등록 필요를
  고지한다. 유지가 불가피할 때만 차선으로 git hook·CI diff 가드(예: lefthook).
  의도적 분기 사본(제네릭화된 마켓플레이스 포크 등)은 단일화·diff 가드 대상이
  아니다 — 미반영 사본으로 보고하고 수동 포팅 필요를 고지한다.
- **레퍼런스 분리**: 트리거 시점에 불필요한 대량 예시·참조 데이터를
  `references/<topic>.md` 로 옮기고 본문엔 언제 읽을지 명시한 포인터 1줄
  (progressive disclosure). 이동이지 삭제가 아니다 — 정보 손실 0.
- **셸 외부화**: 결정적·반복 절차는 `scripts/` 셸 함수로 빼고 본문엔 호출
  1줄만. 프로젝트 기존 컨벤션을 따른다(예: 공개 `claude-*` · 내부 `_claude-*`,
  네트워크 호출은 재시도 래퍼). **동작 패리티 필수** — 전후 LLM 가시
  표면(출력·반환·메시지)이 동일해야 하며 신규 함수는 단위 테스트 추가.
- **훅 흡수**: 결정적 always-run 지시는 대상 스킬과 같은 스코프의 훅
  (`.claude/hooks/<event>.sh` 또는 `~/.claude/hooks/<event>.sh`)으로 옮기고
  같은 스코프 settings.json 의 `hooks` **등록까지가 완료 조건** — 등록 없는
  좀비 스크립트 금지. 훅 실패는 흡수(`exit 0`)하고 stderr 로깅. 흡수된 지시는
  SKILL.md 에서 제거.

## 4단계 — 검증 (동작·트리거 불변 확인)

- md 포맷 게이트가 있으면 실행(prettier 등). 셸 외부화 등 스크립트를 손댔으면
  `shellcheck -x -S warning <손댄 스크립트>` + 번들/프로젝트 테스트 스크립트
  수동 실행(CI 에 안 물린 테스트는 수동 실행이 유일한 게이트).
- **GFM 구조 검증**: 표의 헤더/구분선/행 칸수 일치, 셀 안 `|` 는 `\|` 이스케이프
  — 칸수 불일치 표는 prettier 가 못 고치고 그대로 정렬한다.
- 1단계 인바운드 앵커 재-grep · `description` 은 1단계 트리거 커버리지(의도·시나리오
  집합)가 압축 후에도 전부 매칭되는지 대조(phrasing 변경은 허용, 커버리지 회귀 0) ·
  함수명/시그니처 diff 0 · 사실 주장 SSOT 재대조.
- provenance 잔여 0: `grep -nE "#[0-9]{2,}" <SKILL.md>`.
- 사본 단일화 시: 삭제된 경로 참조 grep 0 + 남은 사본 소비자·라이브 등록 경로
  재확인. 레퍼런스 분리 시: 이동분 전수 parity 대조(항목·필드 단위 — 예: API
  표면이면 메서드/경로/본문) + 포인터 링크 유효.
- 번들 회귀 테스트 있으면 실행 (예: `bash scripts/test-*.sh`).
- 훅 흡수 시: 해당 스코프 settings.json 의 `hooks` 등록 확인
  (`grep -n '"hooks"'`) + 수동 트리거(stdin payload 주입)로 동작 검증.
- **before/after 크기 delta 보고**: 줄·문자·추정 토큰(상시 적재분과 on-demand
  분리 구분).

## 안 하는 것

- 트리거 커버리지 회귀(distinct 매칭 단서 제거 — phrasing 압축이든 삭제든) ·
  인바운드 앵커 헤딩 변경 · 함수 시그니처 변경 · 정보·동작을 줄이는 변경(어떤
  lever 로도 금지).
- 사실 오류를 '정보 불변' 명분으로 보존.
- 등록 없는 좀비 훅 · 패리티 깨지는 외부화 · 소비자 확인 없는 사본 삭제.
- 커밋/PR — 이 스킬 범위 밖(커밋 전용 스킬·수동 절차 소관).
